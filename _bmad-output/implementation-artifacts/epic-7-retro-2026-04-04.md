## Key Insights & Lessons Learned

### Technical Insights

1. **JDBC-only checks are architecturally cleaner than DataFrame checks.** Seven of the eight new check implementations in Epic 7 had no Spark operations. Tests are fast, deterministic, and don't require SparkSession lifecycle management. The `@FunctionalInterface` provider pattern with lambda injection makes these checks trivially testable. This pattern should be the default for all future checks that don't require distributed computation.

2. **The `DqsScoreCheck` timing constraint shapes all Tier 3 checks.** `ClassificationWeightedCheck` cannot read the DQS score because DqsScoreCheck runs last. The resolution — emit a priority multiplier metric as a signal for external consumers — is the right design, but it surprised the dev agent. This constraint should be documented in project-context.md under the Tier 3 check design rules.

3. **Dead public constants and dead method parameters are a recurring first-pass defect.** Three of four stories had code review findings for public constants declared but never referenced in logic, or method parameters declared but never read. This suggests the implementation pattern is: declare the constant "for documentation purposes," then not use it in the actual logic. The fix is a review rule: every public constant must have at least one direct reference in the class body (excluding Javadoc).

4. **Data sensitivity boundary is a new rule class.** Detail metrics (`dq_metric_detail.detail_value`) are JSON payloads consumed by the serve layer and surfaced through the API to the dashboard. Embedding source data values (HDFS path segments, dataset names, column values) into these payloads violates the principle that metrics should contain quality signals, not data values. `OrphanDetectionCheck` was the first violation. The rule must be codified: "detail_value JSON may contain field names, thresholds, scores, and reasons — never source data values like dataset names, path segments, or column content."

5. **Pre-generated ATDD test data for statistical checks requires a mathematical verification pass.** The `InferredSlaCheck` ATDD test used list values that produced stddev ≈ 1.57 (not 2.0) due to incorrect sample construction. The implementation was correct; the test was wrong. The ATDD phase for statistical or mathematical checks should include an explicit verification step: compute the expected output by hand before generating the test assertion.

6. **Executive reporting SQL needs only `v_dq_run_active`.** Monthly LOB rollups, quarterly comparisons, source system accountability, and improvement deltas can all be derived from `DATE_TRUNC('month', partition_date)`, `AVG(dqs_score)`, and `SPLIT_PART(dataset_name, '/', n)` against the existing active run view. No summary tables, no new schema, no DDL changes required. The architecture decision to use a generic metric model pays full dividends at the executive layer.

### Process Insights

7. **100% code review coverage is achievable and was sustained for four consecutive stories.** Epic 5 was 50%, Epic 6 was 60%, Epic 7 was 100%. The mechanism that drove improvement was a concrete retro commitment with a named success criterion. The risk is regression after the epic closes. The 100% gate must be treated as a permanent baseline, not a temporary achievement.

8. **The three project-context.md carries require a story, not a retro commitment.** Four consecutive retrospectives have listed the Python resource management rule, H2 DECIMAL precision note, and FastMCP monitoring note as action items. Four times they have not been done. The pattern is clear: retro action items without story backing don't close. These three items must be incorporated into the first story spec of the next project phase, not left as free-floating commitments.

9. **Paradigm shifts between stories (Java → Python → React) increase dev agent cognitive load.** Stories 7.1–7.3 were structurally similar — JDBC-only Java checks, same provider pattern, same test structure. Story 7.4 required simultaneous familiarity with Python/FastAPI Pydantic models, conftest.py mock behavior, React/MUI component patterns, and Vitest test idioms. The debug log for 7.4 shows one investigation point (mock DB filtering). Future epics mixing multiple languages in a single sprint should ensure story specs include complete per-technology pattern references.

### Team Agreements

**Carry forward from Epic 6:**
- EXPIRY_SENTINEL rule: Never hardcode `9999-12-31 23:59:59` — use `DqsConstants.EXPIRY_SENTINEL`
- Active-record views rule (serve layer): All Python queries target `v_*_active` views
- Spark queries use raw table + `expiry_date = ?` with `EXPIRY_SENTINEL` (not views)
- ATDD RED-phase cleanup required: Remove all `@Disabled` / `# THIS TEST WILL FAIL` annotations before SM review
- Code review completion required: Review Findings section must be present with content

**New agreements from Epic 7:**
- **Data sensitivity boundary rule**: `detail_value` JSON payloads contain field names, thresholds, scores, and reasons — never source data values (dataset names, paths, column content)
- **Tier 3 check timing**: Checks running before `DqsScoreCheck` cannot read the current DQS score; use priority/multiplier signals (ClassificationWeightedCheck pattern) or historical queries (SourceSystemHealthCheck pattern) instead
- **Public constants must be referenced**: Every `public static final` constant must have a direct code reference; documentation-only constants belong in Javadoc, not as declared fields
- **Statistical ATDD requires math verification**: Pre-generated test assertions for statistical checks must include inline derivation of expected values
- **No new no-op checks in DqsJob**: Every new DqCheck registered in `buildCheckFactory()` must use its JDBC constructor, not the no-arg NoOp constructor, regardless of story complexity
