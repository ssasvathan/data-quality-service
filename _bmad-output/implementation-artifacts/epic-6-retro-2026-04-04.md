## Key Insights & Lessons Learned

### Technical Insights

1. **The `DqCheck` interface scales cleanly across very different check types.** A time-based check (no DataFrame), a count check (no providers), a schema-diff check (JDBC baseline), and a multi-pass statistical check (two JDBC providers, Spark aggregation) all registered identically. The architecture decision to keep the interface minimal (`getCheckType()` + `execute()`) paid off across 5 new implementations.

2. **JDBC providers are tested but not wired — this is an architectural gap that must be closed.** `SlaCountdownCheck`, `BreakingChangeCheck`, and `DistributionCheck` each have production-ready JDBC providers with full unit test coverage. But `DqsJob.buildCheckFactory()` uses `NoOp*` providers for all three. The checks produce no metrics in production. This must be resolved before any business value is realized from Epic 6.

3. **"Simple" checks are not exempt from review.** Both unreviewed stories in Epic 6 (6-2, 6-3) have non-trivial hidden complexity: ZeroRowCheck has boundary conditions around null DataFrame and null context that untested code paths could miss; BreakingChangeCheck has a hidden coupling to SchemaCheck's JSON payload format. Simplicity is not an argument against review — it is an argument that the review will be quick.

4. **Distribution check z-score at zero stddev is a subtle production risk.** The review caught that `computeZScore` returning 0.0 when stddev=0 would silently misclassify a shift. In production data, a column that previously had variance and now has zero variance (all same value) would incorrectly show z=0.0 rather than signalling a shift. This is exactly the class of bug that only formal review reveals.

5. **eventAttribute JSON explosion is brittle against non-JSON string columns.** The `spark.read.json()` schema inference is wrapped in try/catch, which is correct. But if a column named `eventAttribute` contains non-JSON strings (e.g., a plain text description), the explosion silently fails and the column is skipped. No metric is written to indicate this. This is a known limitation that should be documented.

### Process Insights

6. **Action items without story attachment don't close.** Five consecutive epic retrospectives have identified the same three items (Python resource management, H2 DECIMAL note, FastMCP monitoring note) for project-context.md. The pattern is clear: without a story-backed dev agent session and an explicit done criterion, items sit in the action list indefinitely. The fix is to attach these to a concrete task in the next sprint, not just relist them.

7. **The code review gate must be binary, not judgment-based.** The current practice allows "SM judgment" to waive review for simple stories. That judgment is consistently wrong — simple stories are exactly where edge cases hide and where discipline flags the hidden coupling or the missing guard. The gate must be: Review Findings section present with content, no exceptions.

8. **Production JDBC wiring should be a done criterion for new checks.** Stories 6.1, 6.3, and 6.4 each documented the JDBC providers as "available for production wiring" but left the wiring as a TODO. This creates a class of "checked in but not functional" code that passes all tests but delivers zero business value. Future check stories must include JDBC wiring as a task, not a TODO.

### Team Agreements

**Carry forward from Epic 5:**
- EXPIRY_SENTINEL rule: Never hardcode `9999-12-31 23:59:59` — use `DqsConstants.EXPIRY_SENTINEL`
- Active-record views rule (serve layer): All Python queries target `v_*_active` views
- Spark queries use raw table + `expiry_date = ?` with `EXPIRY_SENTINEL` (not views)
- ATDD RED-phase cleanup required: Remove all `@Disabled` / `# THIS TEST WILL FAIL` annotations before SM review
- Code review completion required: Review Findings section must be present with content

**New agreements from Epic 6:**
- **No-arg DqCheck constructors should be explicitly documented**: If a no-arg constructor uses a no-op provider, the Javadoc must state "Production use requires wiring via 2-arg/3-arg constructor in DqsJob."
- **"Simple" is not an exemption from code review**: Any story with a DqCheck implementation gets formal review regardless of perceived complexity.
- **JDBC provider wiring is a done criterion**: New checks with JDBC providers must be wired (or explicitly deferred with justification) before story closes as `done`.
