## Key Insights & Lessons Learned

### Technical Insights

1. **Temporal pattern is not a one-time decision — it's a discipline.** Every story must reinforce: `EXPIRY_SENTINEL` constant in Python, literal only in SQL DDL, views for all application queries. This is the core invariant of the platform.

2. **Real Postgres > mocks for constraint testing.** H2 or in-memory databases would have missed the composite unique constraint violations that real Postgres caught. The investment in `docker-compose` from Story 1.1 paid off through all 7 stories.

3. **Append-only DDL is a team agreement, not just a preference.** Any modification to existing SQL statements risks breaking environments that have already run them. Stories followed this discipline; Epic 2 must continue it.

4. **Views as anti-pattern prevention.** `v_*_active` views make the right thing the easy thing — application code cannot accidentally query expired records. This is architectural enforcement, not documentation.

5. **`INSERT...SELECT` with subqueries over hardcoded IDs.** Story 1.7 demonstrated this robustly: FK references via subquery work across environments without assumption about serial sequences.

### Process Insights

6. **Developer checklist gap identified.** Three recurring review findings — hardcoded sentinels, stale TDD comments, cursor management — all avoidable with a pre-review checklist. Action item for Epic 2.

7. **Component independence discipline must be actively maintained.** The rule is clear (no shared resources between components), but as Epic 2 adds Spark Java code, the temptation to reference shared constants grows. Each component is its own island.

8. **Maven Surefire JVM configuration is load-bearing.** The 13 `--add-opens` flags + Java 21 JVM path for tests are non-negotiable for Spark 3.5.0 compatibility. Do not touch Surefire config without understanding why it's there.

### Team Insights

9. **Zero-regression discipline is achievable.** 7 stories, cumulative schema, 0 regressions. This came from: running full test suite before each story completion, append-only DDL, and conftest.py being the single fixture loading point.

10. **"Optional" doesn't mean "skip."** Retrospectives are marked `optional` in sprint-status but have high ROI. First retrospective established 10 actionable lessons — that's a high-leverage investment.

### Team Agreements

- **Temporal pattern rule**: Never hardcode `'9999-12-31 23:59:59'` in Python or Java — always use `EXPIRY_SENTINEL` constant
- **DDL rule**: Append-only — never modify existing SQL statements, only add new ones
- **Views rule**: Application code (Python and Java) queries `v_*_active` views, never raw tables
- **Testing rule**: Test against real Postgres (docker-compose) for all constraint and temporal logic
- **Component rule**: Each dqs-* component owns its own config, dependencies, and build — no sharing
