## Key Insights & Lessons Learned

### Technical Insights

1. **Python resource management requires explicit discipline.** `psycopg2` connections don't use the async context manager that SQLAlchemy provides. Every `get_connection()` call in a try block needs a corresponding `finally: conn.close()` or a `with`-style wrapper. This must be on the checklist.

2. **Cross-language CLI arg wiring is fragile without round-trip tests.** The Python orchestrator passes `--orchestration-run-id` and `--rerun-number` to the Spark CLI, but there's no integration test that verifies the full round trip. Unit mocks catch command construction but not end-to-end parsing. For Epic 4+, consider smoke tests for the full pipeline.

3. **Module-level import patching in pytest is non-obvious.** Story 3.2 completion notes: `run_all_paths` in `test_cli.py` must be patched as `orchestrator.cli.run_all_paths` not `orchestrator.runner.run_all_paths` — because `cli.py` imports it at module level. This is a Python-specific gotcha that trips up developers from Java backgrounds.

4. **isort violations indicate no automated linting gate.** Both Story 3.5 review findings were auto-fixable by `ruff --fix`. If ruff ran in CI or pre-commit, these would never reach review. For Epic 4 with TypeScript/Python, linting gates matter more.

5. **pyproject.toml structure must be validated before first implementation story.** The missing `[build-system]` and `pytest pythonpath` config was only found when Story 3.1 tried to run tests. Component scaffolding validation should be a pre-story task, not a discovery during implementation.

### Process Insights

6. **The dev checklist pattern keeps failing for the same reason.** We commit to creating `dev-checklist.md` in the retro. The SM writes it as an action item. The first dev story starts without checking whether it exists. The cycle repeats. The fix: SM creates the file in this retro session before closing — not as a future action item.

7. **Code review is a gate, not a suggestion.** Stories 3.1 and 3.2 shipped without code review records. The review process catches real issues (7 findings across 3.3 and 3.5). Every story must have Review Findings filled before status can be set to `done` — even if the finding is "0 findings."

8. **ATDD red-phase is the reliability foundation.** Stories with ATDD pre-written tests (3.1–3.5) had clean first-pass implementations that drove all ACs to green. The test suite acts as a specification that the implementation must satisfy — not an afterthought.

### Team Insights

9. **Language transition competency.** The team went from pure Java (Epic 2) to Python (dqs-orchestrator) in a single story. Python idioms (type hints, dataclasses, `subprocess.run`, `smtplib`) were applied correctly. The Epic 2 retro note about "testing approach changes from JUnit/H2 to pytest/tmp_path" was absorbed.

10. **Zero-regression discipline held at scale.** With 80 Python + 177 Java tests and 5 stories of incremental extension, no story broke any prior story's tests. The discipline of running `uv run pytest -v` and `mvn test` as a quality gate is working.
