## Key Insights & Lessons Learned

### Technical Insights

1. **ADDITIVE architecture constraint eliminates regression risk.** Defining the MCP layer as purely additive (no existing routes touched) meant zero regression risk across 149 tests. Architectural constraints that reduce blast radius are worth making explicit in story specs.

2. **Direct function testing pattern scales for MCP tools.** Calling `query_failures(db=mock_db)` directly (bypassing FastMCP's call machinery) is simpler and faster than ASGI-level unit tests. This pattern should be documented as the standard for all future MCP tool tests.

3. **SQL duplication is correct for module isolation.** MCP tools copying SQL from route files rather than importing is the right architectural call. The alternative — importing from `routes/summary.py` — would couple the MCP module to the API layer, creating a maintenance dependency. Document this as a deliberate choice, not a quality gap.

4. **FastMCP deprecation requires proactive monitoring.** `exclude_args` was deprecated in FastMCP 2.14 and is still functional in 3.2.0. If 4.x removes it, all three tools in `tools.py` need updating. Add a note to project-context.md: "Monitor FastMCP changelog for `exclude_args` removal before upgrading past 3.x."

5. **Story spec SQL examples are the highest-ROI documentation.** Both stories shipped on first pass, attributed by the dev agent to the verbatim SQL patterns in Dev Notes. Pre-specifying exact SQL (not pseudo-SQL) eliminates the largest source of implementation variance.

### Process Insights

6. **project-context.md is the canonical rules source — the checklist is not a substitute.** Two Epic 4 action items (Python resource management, H2 DECIMAL) ended up in dev-checklist.md but not project-context.md. Story spec writers reference project-context.md, not the checklist. Rules that aren't in project-context.md don't reliably reach implementation.

7. **'Clean first pass' is not a review.** Story 5.1's clean implementation is commendable, but the absence of a Review Findings section means no second pair of eyes verified the SQL column names, error handling paths, or edge cases. The pre-existing `m.run_id` bug in 5.1 was only caught during 5.2's review.

8. **Review findings in 5.2 had direct business impact.** The `_normalize_time_range()` patch fixed a user-visible bug (invalid time_range rendered literally in response). That's the review process working as intended. Epic 5 reinforces: reviews find real bugs.
