## Key Insights & Lessons Learned

### Technical Insights

1. **ATDD RED-phase comment cleanup must be a done criterion.** 13/15 stories had stale `// THIS TEST WILL FAIL` / `// RED PHASE:` comments in test files. These are implementation artifacts from TDD and should be removed before review — not left as permanent documentation noise.

2. **Floating Promise `void`-prefix is a TypeScript-specific discipline.** The `no-floating-promises` rule flags unhandled Promise returns in event handlers. This is different from Java's synchronous patterns. The pattern `void someAsyncCall()` must be on the pre-review checklist for TypeScript stories.

3. **Component abstraction boundaries held at epic scale.** `DqsScoreChip`, `TrendSparkline`, `DatasetCard`, and `DatasetInfoPanel` were built once and reused identically across all consumer views. The decision to NOT mix rendering with data fetching inside these components (they receive props only) was the right architectural call.

4. **Reference data resolution must be server-side.** Client-side LOB name resolution would cause N+1 lookups per dataset row. The 12h TTL cache in Story 4.5 with a graceful "N/A" fallback is the right pattern and should be documented as a project rule.

5. **React Router 7 deep-linking requires query key alignment.** Shareable URLs work only when React Query cache keys include the URL parameters. The `timeRange` in every query key was the right decision — it enables browser back/forward, bookmarking, and shared links with correct data.

6. **Skeleton screens are a component-level responsibility.** Every data-fetching component implements its own skeleton that matches target layout dimensions. This prevents layout shift on data arrival and is now an established pattern for Epic 5.

### Process Insights

7. **The dev-checklist cycle is broken at the system level.** Three epics, three retro commitments, zero file created. The solution is not another action item — it is creating the file in this session and adding its existence as a pre-story gate in the sprint-planning workflow.

8. **Code review coverage improved but is not enforced.** Going from 3/5 (60%) to 10/15 (67%) shows improvement, but the 5 unreviewed stories include the most complex story in the epic (4.12). The gate rule needs to be machine-enforced (e.g., SM checks sprint-status for review section presence before setting `done`).

9. **Stale ATDD comments need an explicit cleanup step in story completion.** The ATDD workflow creates "THIS TEST WILL FAIL" comments as scaffolding. These should be swept before the SM review phase — not after. Add to done criteria.
